= 💡 Solution: Scenario 6 — ALIA Lightspeed issues

This page provides the detailed solution for the issue presented in Scenario 6, which involves an incorrect Large Language Model (LLM) configuration for Ansible Lightspeed Intelligent Assistant (ALIA).

---

== 🛑 Problem: Invalid Large Language Model (LLM) configuration

=== Diagnosis

The *Ansible Lightspeed Intelligent Assistant (ALIA)* service was unable to initialize because the `chatbot-configuration-secret` contained two misconfigurations:

* The model was set to one not available in this environment (`codellama-7b-instruct` instead of `{alia_model}`)
* The API token was invalid (`sk-invalid-token-for-troubleshooting` instead of the correct LiteMaaS key)

The pod logs for the `lightspeed-api` and `chatbot` pods confirm the model connection or authentication failure.

=== 🛠️ Resolution: Updating the chatbot secret

The configuration for the LLM is stored within the `chatbot-configuration-secret`. The fix is to edit this secret and correct both the model name and the API token.

==== 1. Update the chatbot configuration secret

In the *OpenShift Console*, navigate to *Workloads → Secrets* in your namespace. Find `chatbot-configuration-secret`, open it and click *Edit*.

Update both `chatbot_model` and `chatbot_token` to the correct values:

[source,yaml,subs="+attributes"]
----
data:
  chatbot_model: {alia_model}  # <--- CORRECT VALUE
  chatbot_token: {alia_token}  # <--- CORRECT VALUE
----

[IMPORTANT]
====
Make sure you update both keys. If using the *Key/Value* editor, the console handles base64 encoding automatically. If editing raw YAML, the values must be base64-encoded.
====

==== 2. Force reconciliation

The *Lightspeed Operator* must restart to pick up the changes from the secret.

In the *OpenShift Console*, navigate to *Workloads → Pods* and select your namespace. Find the `lightspeed-api` pod, click the three-dot menu on the right, and select *Delete Pod*. Repeat for the `lightspeed-chatbot-api` pod. Both will restart automatically and pick up the corrected configuration.

==== 3. Verification

Monitor *Workloads → Pods* until the Lightspeed pods are running and ready. Then log back into the AAP UI; the ALIA chat icon should reappear in the top navigation bar and respond to a test query.
